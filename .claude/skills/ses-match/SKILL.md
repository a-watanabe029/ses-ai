# /ses-match — 要員→案件マッチング

CLAUDE.md §実現2 に対応するSkill。v0は方向A（要員→案件）のみ。しきい値表による機械的フィルタではなく、**Claudeが都度判断**する。

## 対象範囲（2モード）

- **対象指定モード**: プロンプトで特定の要員名が挙げられた場合、その1名だけを対象にする。
- **一括モード**: 特定の要員名が挙げられない場合（Routineでの無人実行を含む）、Google Drive「稼働中」フォルダ直下の**全人員フォルダ**を対象に、各人員を**1名ずつ順に**（並列化しない）マッチングする。許容条件プロンプトの個別適用はできない。

## 前提

- 要員プロフィールはGoogle Drive（共有アイテム＞02.セールスマーケチーム＞99.営業支援(テスト)＞`稼働中`。フォルダID=`PROFILE_FOLDER_ID`）に用意されていること。`稼働中`直下に人員フォルダ（フォルダ名＝要員名）があり、各人員フォルダの中に`<要員名>.md`が1つ置かれている構成。**ローカルの`要員/<名前>.md`は参照しない**（Routine等リモート実行環境ではローカルフォルダを確認できないため）。未整備なら本人へ確認、または既存スキルシート等の資料があればテンプレート `要員/_テンプレート.md` の内容をアップロードする
- `.env` に `SHEET_ID`・`PROFILE_FOLDER_ID` 設定済み、OAuth 設定済み（`secrets/credentials.json`・`secrets/token.json`）、`.venv` 構築済み

## 手順

### 1. 案件台帳を読む（要員ごとの差分・DECISIONS §9）

対象要員 `<名前>` について、**前回マッチ以降に台帳へ入った案件だけ**を読む:

```
.venv/bin/python scripts/read_sheet.py --person <名前>        # macOS/Linux
.venv\Scripts\python.exe scripts\read_sheet.py --person <名前>   # Windows
```

- `_state` の `match_cursor.<名前>`（`ingested_at` 到達点）**以降の案件だけ**をJSONで返す。カーソル未設定（初回）は全件（新規要員のオンボーディング）。
- **stderr に `[cursor] high_water=<ISO>` を出力する。この値を控えておき、手順5の `--advance-cursor` に渡す**（読取自体はカーソルを進めない＝副作用なし）。
- stdout側はヘッダー行をキーにしたdictのリスト（列構成の変更に自動追従）。シェルの`>`リダイレクトではなく、Pythonスクリプト側で`encoding="utf-8"`のファイル書き込みを行い、Readツールで読むこと（文字化け防止）。※ `[cursor]` 行はstderrなのでコマンド出力にそのまま出る。
- **プロフィールや許容条件を変えて出し直したいときは `--person` の代わりに `--full`**（カーソル無視で全件）。この場合カーソルは動かさない（手順5で `--advance-cursor` を渡さない）。
- 件数が多い場合（初回の全件など）、直接読んで評価するかサブエージェントに委任するかは件数次第で判断してよいが、マッチング判断は業務理解が要るため基本は自分（メインの会話）で評価する方が精度が高い。

### 2. 要員プロフィールの取得

**対象指定モード**（要員名が指定されている場合）:

```
.venv/bin/python scripts/read_profile.py <名前>       # macOS/Linux
.venv\Scripts\python.exe scripts\read_profile.py <名前>  # Windows
```

- 人員フォルダまたは`<名前>.md`が無ければ`FileNotFoundError`（§エラー・例外を参照し中断）。
- その場で許容条件のプロンプト（例:「経験3年だが優秀なので4年要件まで可」「隣県も可」「単価60万以上なら可」）が与えられていれば、本人条件に対する上書き・追加条件として適用する。

**一括モード**（要員名が指定されていない場合）:

```
.venv/bin/python scripts/read_profile.py       # macOS/Linux（引数なし）
.venv\Scripts\python.exe scripts\read_profile.py  # Windows（引数なし）
```

- `稼働中`直下の全人員フォルダを走査し、`[{"person_name": "...", "content": "..."}, ...]`のJSON配列を返す。
- 人員フォルダ内に`<人員名>.md`が無いものは、エラーにせずスクリプト側が自動的にスキップし、`[skip] <人員名>: ...`という行をstderrに出力する。この場合その人員は対象外として扱い、他の対象者の処理を続ける。
- 許容条件プロンプトの個別適用はできない（一括実行のため）。
- 得られた対象者リストを**1名ずつ順に**（並列化しない）以下の手順3〜5にかける。

### 3. 評価

**評価軸**（★=対称・単価と経験は非対称）:
- 必須スキル充足
- 経験年数（許容込み）
- 勤務地・居住地（隣県許容込み）
- 単価（**案件=上限予算 vs 人材=下限希望**。区間の重なりで判定）
- リモート
- 開始時期
- 商流

**判定ルール**:
- 国籍・年齢・性別は**機械的な足切りにしない**（職業安定法・労働施策総合推進法9条）。案件側に制限記載があっても、参考情報として懸念欄に記すに留め、適法性要確認の警告を添える
- 適合度は ◎（強い一致）／○（好条件だが一部懸念）／△（部分一致・要確認点が多い）の3段階
- 明確に不一致（言語・職種・経験年数が大きく乖離等）な案件は対象外としてよい。ただし「惜しい」候補（スキルは合うが勤務地がネック等）は積極的に△として残す（機械的除外よりは人間の判断に委ねる）
- 適合理由・懸念は、要員のプロフィール（経歴・希望条件）の具体的な根拠と紐付けて記述する（例:「PJ4の出荷管理システム経験がこの物流案件のドメインと一致」）
- **適用した許容条件は適合理由に明記**する（再現性のため。結果行に残る）

### 4. マッチ結果の書き出し

`data/matches/<名前>.jsonl` に1行1案件で追記する。既存ファイルがあれば末尾に追記（`write_match.py`側で`row_key`による重複排除が行われるため、同じ案件を再掲しても害はない）。

各レコードのフィールド:
```
row_key（案件台帳のrow_key＝messageId）, 適合度（◎/○/△）, 案件名, 単価, 勤務地,
リモート, 商流, 必須スキル, 適合理由, 懸念, リンク（案件メールリンク）
```

### 5. 本人タブへ反映（＋カーソル前進）

```
.venv/bin/python scripts/write_match.py <要員名> --advance-cursor <手順1のhigh_water>       # macOS/Linux
.venv\Scripts\python.exe scripts\write_match.py <要員名> --advance-cursor <手順1のhigh_water>  # Windows
```

- 本人タブ（タブ名＝要員名）の列: `記載日 / 適合度 / 案件名 / 単価 / 勤務地 / リモート / 商流 / 必須スキル / 適合理由 / 懸念 / ステータス / 案件row_key(messageId) / リンク`
- `記載日`＝実行日、`ステータス`＝未指定なら「未対応」をスクリプトが自動設定する
- タブが存在しなければ見出し行付きで自動作成する
- **既存の`案件row_key(messageId)`列で重複回避**（再実行は新規のみ・既存行の手動ステータスは保持）
- **`--advance-cursor` に手順1で控えた `high_water` を渡す**と、追記成功後に `match_cursor.<名前>` を前進させる（overlap 10分を差し引いて保存）。次回以降その要員はこの差分を再評価しない（DECISIONS §9）。順序＝**追記→カーソル前進**（途中失敗は次回再スキャンで回収・messageId dedup が二重行を防ぐ）。
- **差分にマッチが0件でも `--advance-cursor` 付きで実行する**（評価済みの差分を再評価しないため）。結果ファイルが無い/空でもカーソルだけ前進する。
- **`--full` で全件再マッチしたときは `--advance-cursor` を渡さない**（カーソルを動かさない）。
- **注意**: 手動でタブを事前作成していた場合、見出し行が無いとデータが1行目から書かれてしまうバグが過去に発生した（`_ensure_tab`で見出し有無を確認するよう修正済みだが、念のため実行後にタブの1行目が見出しになっているか確認する）

## 完了後の報告（画面表示ルール・2026-07-03決定）

**一括モード**では、対象者ごとの報告の前に全体サマリー（対象N名／スキップM名とその人員名）を示す。

対象者ごとに、冒頭で必ず ◎/○/△ それぞれの件数サマリーを示す。詳細ブロックの表示範囲は **◎の件数**で切り替える（コンテキスト消費とのバランスのため）。

- **◎が10件以内**: ◎と○を1件1ブロックでフル表示。△は件数のみ＋「詳細は本人タブでご確認ください」
- **◎が10件超**: ◎のみ1件1ブロックでフル表示。○・△は件数のみ

各ブロックのフォーマット（CLAUDE.md準拠）:
```
### <適合度> <案件名>
- 単価: XX万円 / 勤務地: XXX / リモート: XXX / 商流: XXX
- 必須スキル: XXX
- 適合理由: XXX（許容条件を適用した場合は明記）
- 懸念・要確認: XXX
- リンク: <案件メールリンク>
```

## エラー・例外

- **対象指定モード**: 人員フォルダまたは`<名前>.md`がDriveに無い（`read_profile.py <名前>`がFileNotFoundError）→ マッチングを実行せず、本人確認またはテンプレート（`要員/_テンプレート.md`）からの作成を案内。
- **一括モード**: 人員フォルダに`<人員名>.md`が無いものがあっても**エラーにせずその要員だけスキップ**し、他の対象者の処理は続行する。完了報告のサマリーにスキップ件数・対象者名を含める。
- マッチ結果ファイル無しで `write_match.py` を実行 → FileNotFoundError（先に評価・書き出しを行う）。ただし `--advance-cursor` 指定時は「差分にマッチ0件」として追記0件で扱いカーソルのみ前進する。
- 案件台帳が空／差分が0件（前回マッチ以降に新着案件が無い）→ 評価対象なし。カーソルは動かさず「新着案件はありません」と案内してよい。全件を見直したい場合は `--full` を案内。

## スコープ外（backlog / 将来）

- 方向B（案件→人材・`マッチ結果` タブ）→ 将来。
- `person_id` 主キー・行ロック（TOCTOU回避）→ backlog（v0 はタブ名＝要員名）。
- **要員ごと差分カーソル（②・`ingested_at` 基準）は実装済み**（DECISIONS §9）。鮮度TTL（①・`received_at` 基準で古い案件を除外）・粗フィルタ緩和（隣接県表）・スキル正準辞書 → backlog。
- 一括モードの単一読取＋カーソル別パーティション化（今は要員ごとに read_sheet を呼ぶ）→ backlog。
- 単一マッチ結果タブへの移行（人員タブ>30）→ backlog。
