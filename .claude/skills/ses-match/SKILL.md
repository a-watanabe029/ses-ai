# /ses-match — 要員→案件マッチング

CLAUDE.md §実現2 に対応するSkill。v0は方向A（要員→案件）のみ。しきい値表による機械的フィルタではなく、**Claudeが都度判断**する。

## 対象範囲（2モード）

- **対象指定モード**: プロンプトで特定の要員名が挙げられた場合、その1名だけを対象にする。
- **一括モード**: 特定の要員名が挙げられない場合（Routineでの無人実行を含む）、Google Drive「稼働中」フォルダ直下の**全人員フォルダ**を対象に、各人員を**1名ずつ順に**（並列化しない）マッチングする。**各人の md `許容条件` は自動で効く**（F-1）が、その場の許容条件プロンプトの個別適用はできない。

**許容条件（+a）は二層（F-1）**: ①恒常＝要員プロフィール（md）の `- 許容条件:` 行（本人と一体・全モードで自動適用）／②都度＝その場の許容条件プロンプト（対象指定モードのみ。mdの許容条件・本人条件への上書き/追加）。

## 前提

- 要員プロフィールはGoogle Drive（共有アイテム＞02.セールスマーケチーム＞99.営業支援(テスト)＞`稼働中`。フォルダID=`PROFILE_FOLDER_ID`）に用意されていること。`稼働中`直下に人員フォルダ（フォルダ名＝要員名）があり、各人員フォルダの中に`<要員名>.md`が1つ置かれている構成。**ローカルの`要員/<名前>.md`は参照しない**（Routine等リモート実行環境ではローカルフォルダを確認できないため）。未整備なら本人へ確認、または既存スキルシート等の資料があればテンプレート `要員/_テンプレート.md` の内容をアップロードする
- `.env` に `SHEET_ID`・`PROFILE_FOLDER_ID` 設定済み、OAuth 設定済み（`secrets/credentials.json`・`secrets/token.json`）、`.venv` 構築済み

## 手順

**共通方針（差分がデフォルト・評価は並列/書込みは直列・DECISIONS §9）**: 各対象者について「**その人の差分案件 × その人の本人条件**」を評価し、**その人のタブ**へ追記して**その人のカーソル**を進める（人ごとに独立）。**一括モードは稼働中フォルダの全員を自動対象**（人員を名指ししない＝人が増減してもコマンド不変。新規要員はカーソル未設定なので初回だけ全件＝取りこぼしなし）。

**一括モードの並列/直列の切り分け（重要）**: **評価（マッチング判定）は要員ごとに並列**（1要員=1サブエージェント。手順2）で行い、**Sheets（本人タブ・`人員一覧`）と `_state`（カーソル）への書込みは本体で1名ずつ直列**（手順3）に行う。共有タブ・`_state` は「タブ丸ごと read-modify-write」で更新されるため、並列に書くと lost update でカーソル等を取りこぼす（→ [[project_state_write_not_concurrent_safe]]）。**サブエージェントは評価と `data/matches/<名前>.jsonl` の書き出しのみで、Sheets/`_state` には一切触らせない**。全件で見直したいとき（スキル/プロフィール編集後）は「### 5. 全件で再マッチ」を参照。

### 1. 差分入力の取得（モードで分岐）

**一括モード**（要員名の指定なし・Routineの無人実行・**既定**）: 稼働中フォルダの全員ぶんを**1コマンド**で束ね、**評価入力を要員ごとのファイルに分割**する（`--emit-inputs`）:

```
.venv/bin/python scripts/match_prepare.py --emit-inputs       # macOS/Linux（--full で全員全件）
.venv\Scripts\python.exe scripts\match_prepare.py --emit-inputs  # Windows
```

- `--emit-inputs`（既定 DIR=`data/matches/_input`）は、delta>0 の各要員の入力（`profile`＋`delta`案件）を `data/matches/_input/<名前>.json` に書き出し、**stdout は delta を含まない軽量マニフェスト** `{"high_water": "<ISO>", "targets": [{"person_name","cursor","delta_count","input_file"}, ...]}` を返す（並列サブエージェントに1要員=1ファイルで渡し、本体コンテキストに大量の delta を載せないため）。
- `input_file` が **null**（delta 0件）の要員は**評価をスキップ**し、手順3のカーソル前進のみ行う。`high_water` は**全員共通**で手順3のカーソル前進に使う。
- `<人員名>.md` が無い人員フォルダは `[skip]` を stderr に出してスキップ（対象外・他は続行。報告に含める）。
- 許容条件プロンプトの個別適用はできない（一括のため）。
- stdout(マニフェスト) は `>` リダイレクトでなく `encoding="utf-8"` のファイル書き込み＋Readで読む（cp932文字化け防止）。`[skip]`/`[match_prepare]` 行は stderr。

**対象指定モード**（特定の要員名が指定された／許容条件を個別適用したい）: その1名だけ:

```
.venv/bin/python scripts/read_profile.py <名前>            # 本人条件
.venv/bin/python scripts/read_sheet.py --person <名前>       # 差分案件（stderr に [cursor] high_water=）
```
（Windows: `.venv\Scripts\python.exe scripts\read_profile.py <名前>` ／ `.venv\Scripts\python.exe scripts\read_sheet.py --person <名前>`）

- `<名前>.md` が無ければ `FileNotFoundError`（§エラー・例外）。
- md 本文の `許容条件` 行は恒常的な +a として読む（F-1）。加えて許容条件プロンプト（例:「経験3年だが優秀なので4年要件まで可」「隣県も可」「単価60万以上なら可」）があれば本人条件・mdの許容条件へ上書き・追加適用する。
- **stderr の `[cursor] high_water=<ISO>` を控える**（手順3で使う）。読取自体はカーソルを進めない。

### 2. 評価（一括＝要員ごと並列サブエージェント／対象指定＝インライン）

**評価は Sheets/`_state` に触れない**。各要員の `data/matches/<名前>.jsonl` を書き出すところまでが評価で、Sheets への反映は手順3（直列）で本体が行う。

**一括モード（並列）**: マニフェストの `input_file` が **非 null の各要員について、1要員=1サブエージェントを並列に起動**する（`delta_count` が大きい要員が居ても互いに独立して評価できる）。各サブエージェントに与える指示:
- `data/matches/_input/<名前>.json`（`profile`＋`delta`案件＋`high_water`）を **Read**。
- その人の各案件を本人条件（`profile`。md 内の `許容条件` 行を恒常 +a として適用・F-1）と「### 4. 評価基準」で照合し ◎/○/△ を判定。
- 結果を `data/matches/<名前>.jsonl` に1行1案件で書き出す（下記フォーマット）。**Sheets・`_state`・`write_match.py` には一切触れない**。
- サブエージェントのプロンプトには「### 4. 評価基準」（評価軸・判定ルール・許容条件の扱い・国籍/年齢/性別の非足切り）と出力フォーマットを**省略せず**明記する（業務理解を要するため、判定精度を落とさないよう基準を渡し切る）。
- **全サブエージェントの完了を待って（join）から**手順3へ進む。`input_file` が null（差分0件）の要員は評価不要（手順3でカーソルのみ前進）。
- **少数のときはインライン可**: 評価対象（`input_file` 非 null）が1〜2名、または各人の `delta_count` が小さいときは、サブエージェントを起動せず本体でまとめて評価してよい（spawn オーバーヘッドの回避）。並列化の狙いは「複数人ぶんの評価を同時に進める」ことなので、対象が薄いときは効果より overhead が勝る。**書込み（手順3）は人数に関わらず常に直列**。

**対象指定モード（インライン・1名）**: 本体（メインの会話）が `read_sheet.py --person` の差分案件を本人条件（＋許容条件プロンプト）と照合して直接評価する（1名なので並列化しない。業務理解を活かせる）。

**書き出しフォーマット**（`data/matches/<名前>.jsonl`・1行1案件・`write_match.py` 側で `row_key` dedup 済み）:
```
row_key（案件台帳のrow_key＝messageId）, 適合度（◎/○/△）, 案件名, 単価, 勤務地,
リモート, 商流, 必須スキル, 適合理由, 懸念, リンク（案件メールリンク）
```

### 3. 直列書込＝本人タブへ反映＋カーソル前進（全評価の完了後・1名ずつ順に）

**一括モードは、全サブエージェントの評価完了（join）を待ってから**、対象者を**1名ずつ順に**（並列化しない）この書込みを行う。**ここが唯一の Sheets／`_state` 書込み地点**（本人タブ append・`人員一覧` upsert・カーソル前進をまとめて直列に行うので、共有タブ・`_state` への同時書込みが無く lost update しない → [[project_state_write_not_concurrent_safe]]）。

```
.venv/bin/python scripts/write_match.py <名前> --advance-cursor <high_water>       # macOS/Linux
.venv\Scripts\python.exe scripts\write_match.py <名前> --advance-cursor <high_water>  # Windows
```

- **`high_water`**: 一括モード＝`match_prepare` の `high_water`（全員共通）／対象指定モード＝手順1で控えた read_sheet の値。
- 本人タブ（タブ名＝要員名）の列: `記載日 / 適合度 / 案件名 / 単価 / 勤務地 / リモート / 商流 / 必須スキル / 適合理由 / 懸念 / ステータス / 案件row_key(messageId) / リンク`。`記載日`＝実行日・`ステータス`未指定は「未対応」を自動設定。タブが無ければ見出し付きで自動作成。
- **既存の`案件row_key(messageId)`列で重複回避**（再実行は新規のみ・既存行の手動ステータスは保持）。
- **名簿 `人員一覧` も自動更新**: 本人タブ書込のたびに `名前 / タブリンク / 件数` を upsert。新規要員は行追加（マッチ0件でも 件数0 で登載）、既存要員は件数のみ更新。→ 稼働中フォルダへ要員を追加して一括モードを回せば、本人タブと名簿の両方が自動で用意される。
- 追記成功後に `match_cursor.<名前>` を `high_water` へ前進（overlap なし＝カーソルが最新行と一致し、新着が無ければ次回差分0）＝次回以降その要員は差分だけ評価（DECISIONS §9）。順序＝**追記→前進**（途中失敗は次回再スキャンで回収・messageId dedup が二重行を防ぐ）。
- **差分にマッチが0件でも `--advance-cursor` 付きで実行する**（評価済みの差分を再評価しないため。結果ファイルが無い/空でもカーソルだけ前進）。
- **注意**: 手動でタブを事前作成していた場合、見出し行が無いとデータが1行目から書かれるバグが過去にあった（`_ensure_tab`で確認するよう修正済みだが、実行後にタブ1行目が見出しか確認する）。

### 4. 評価基準

**評価軸**（★=対称・単価と経験は非対称）:
- 必須スキル充足
- 経験年数（許容込み）
- 勤務地・居住地（隣県許容込み）
- 単価（**案件=上限予算 vs 人材=下限希望**。区間の重なりで判定）
- リモート
- 開始時期
- 商流

**判定ルール**:
- 国籍・年齢・性別は**機械的な足切りにしない**（職業安定法・労働施策総合推進法9条）。案件側に制限記載があっても、参考情報として懸念欄に記すに留め、適法性要確認の警告を添える
- 適合度は ◎（強い一致）／○（好条件だが一部懸念）／△（部分一致・要確認点が多い）の3段階
- 明確に不一致（言語・職種・経験年数が大きく乖離等）な案件は対象外としてよい。ただし「惜しい」候補（スキルは合うが勤務地がネック等）は積極的に△として残す（機械的除外よりは人間の判断に委ねる）
- 適合理由・懸念は、要員のプロフィール（経歴・希望条件）の具体的な根拠と紐付けて記述する（例:「PJ4の出荷管理システム経験がこの物流案件のドメインと一致」）
- **許容条件の出所（F-1）**: プロフィール（md）の `許容条件` 行＝恒常的な +a として常に適用（一括モードでも各人ぶんが効く）。対象指定モードではさらにプロンプト指定を上書き/追加。どちらも機械的しきい値でなく判断材料
- **適用した許容条件は適合理由に明記**する（再現性のため。結果行に残る）

### 5. 全件で再マッチ（スキル/プロフィール編集後）

差分は既定。**評価ロジックや本人条件を変えたら、その分を全件で再評価**する。全件は「カーソルを消す＝次回の通常ランがそのまま全件になる」で行う（別モードは持たない）:

- **スキル（評価ロジック）を編集した → 全員を再評価**:
  `.venv/bin/python scripts/state.py match-cursor reset --all` を実行 → その後は通常どおり手順1〜3を回す（全員カーソル未設定＝全件→再評価→カーソル再設定）。
- **1人のプロフィールを編集した → その人だけ再評価**:
  `.venv/bin/python scripts/state.py match-cursor reset <名前>`。
- **現在のカーソル確認**: `.venv/bin/python scripts/state.py match-cursor list`。
- **その場で1回だけ全件を見たい（カーソルを動かさない）**: 一括モードは手順1を `match_prepare.py --full`（各 target が全件になる）、対象指定モードは手順1を `read_sheet.py --full`。いずれも手順3で `--advance-cursor` を付けない（カーソルを動かさない）。
- 注意: `write_match.py` は追記のみ（messageId dedup）なので、全件再マッチしても**既存タブ行の適合度は上書きされず、新規に該当した案件が追加されるだけ**（upsert は backlog）。

## 完了後の報告（画面表示ルール・2026-07-03決定）

**一括モード**では、対象者ごとの報告の前に全体サマリー（対象N名／スキップM名とその人員名）を示す。

対象者ごとに、冒頭で必ず ◎/○/△ それぞれの件数サマリーを示す。詳細ブロックの表示範囲は **◎の件数**で切り替える（コンテキスト消費とのバランスのため）。

- **◎が10件以内**: ◎と○を1件1ブロックでフル表示。△は件数のみ＋「詳細は本人タブでご確認ください」
- **◎が10件超**: ◎のみ1件1ブロックでフル表示。○・△は件数のみ

各ブロックのフォーマット（CLAUDE.md準拠）:
```
### <適合度> <案件名>
- 単価: XX万円 / 勤務地: XXX / リモート: XXX / 商流: XXX
- 必須スキル: XXX
- 適合理由: XXX（許容条件を適用した場合は明記）
- 懸念・要確認: XXX
- リンク: <案件メールリンク>
```

## エラー・例外

- **対象指定モード**: 人員フォルダまたは`<名前>.md`がDriveに無い（`read_profile.py <名前>`がFileNotFoundError）→ マッチングを実行せず、本人確認またはテンプレート（`要員/_テンプレート.md`）からの作成を案内。
- **一括モード**: 人員フォルダに`<人員名>.md`が無いものがあっても**エラーにせずその要員だけスキップ**し、他の対象者の処理は続行する。完了報告のサマリーにスキップ件数・対象者名を含める。
- マッチ結果ファイル無しで `write_match.py` を実行 → FileNotFoundError（先に評価・書き出しを行う）。ただし `--advance-cursor` 指定時は「差分にマッチ0件」として追記0件で扱いカーソルのみ前進する。
- 案件台帳が空／差分が0件（前回マッチ以降に新着案件が無い）→ 評価対象なし。カーソルは動かさず「新着案件はありません」と案内してよい。全件を見直したい場合は `--full` を案内。

## スコープ外（backlog / 将来）

- 方向B（案件→人材・`マッチ結果` タブ）→ 将来。
- `person_id` 主キー・行ロック（TOCTOU回避）→ backlog（v0 はタブ名＝要員名）。
- **要員ごと差分カーソル（②・`ingested_at` 基準）は実装済み**（DECISIONS §9）。鮮度TTL（①・`received_at` 基準で古い案件を除外）・粗フィルタ緩和（隣接県表）・スキル正準辞書 → backlog。
- 一括モードの単一読取＋カーソル別パーティション化（`match_prepare.py`）→ **実装済み**（2026-07-08）。台帳1回読みで稼働中全員ぶんの差分を束ねる。
- 一括モードの**評価並列化（要員ごとサブエージェント）＋書込み直列**（`match_prepare.py --emit-inputs` で1要員=1入力ファイル→並列評価→本体が `write_match.py` を1名ずつ）→ **実装済み**（2026-07-10）。評価は並列・共有リソース（Sheets/`_state`）書込みは直列で lost update を回避（[[project_state_write_not_concurrent_safe]]）。書込みまで並列化する場合の per-cell `_state` 更新・楽観ロックは backlog。
- 単一マッチ結果タブへの移行（人員タブ>30）→ backlog。
